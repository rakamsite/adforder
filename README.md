# Filter Inquiry Portal

## معرفی افزونه

**Filter Inquiry Portal** یک افزونه اختصاصی وردپرس برای راه‌اندازی پنل فارسی و راست‌به‌چپ استعلام فیلتر است. این افزونه به‌صورت مرحله‌ای توسعه داده می‌شود تا در نهایت مشتری بتواند با شماره موبایل وارد شود، پروفایل خود را تکمیل کند، محصولات را جستجو کند، درخواست استعلام ثبت کند و وضعیت درخواست‌های خود را از داخل داشبورد پیگیری کند.

## هدف پروژه

هدف پروژه ساخت یک پورتال سبک، قابل‌توسعه و امن برای مدیریت فرایند استعلام فیلتر در وردپرس است؛ به شکلی که تنظیمات اصلی از پیشخوان وردپرس کنترل شود و توسعه فازهای بعدی بدون بازنویسی هسته افزونه امکان‌پذیر باشد.

## وضعیت فعلی: Phase 2

در **Phase 2** ورود و ثبت‌نام با شماره موبایل و کد یک‌بارمصرف فعال شده است. شورت‌کد `[filter_portal_login]` فرم دریافت شماره موبایل، تولید OTP، نمایش مرحله ورود کد، ساخت/ورود کاربر و ریدایرکت پایه به تکمیل پروفایل یا داشبورد را انجام می‌دهد. ارسال واقعی پیامک هنوز عمداً انجام نمی‌شود و کد فقط به‌صورت mock برای توسعه ذخیره و در شرایط امن توسعه نمایش داده می‌شود. ثبت درخواست، جستجوی محصول WooCommerce، داشبورد نهایی و اتصال واقعی sms.ir همچنان برای فازهای بعدی باقی مانده‌اند.

## امکانات پیاده‌سازی‌شده در فاز 0، 1، 1.5، 2، 4 و 6

### Phase 0

- ساختار اصلی افزونه و فایل bootstrap.
- کلاس Singleton اصلی `FIP_Plugin`.
- ماژول‌بندی اولیه در پوشه `includes/`.
- ثبت اولیه assetها بدون enqueue سراسری.
- قالب‌های placeholder اولیه در پوشه `templates/`.
- فایل uninstall محافظت‌شده بدون حذف داده.

### Phase 1

- صفحه تنظیمات افزونه در پیشخوان با عنوان منوی **استعلام فیلتر**.
- ذخیره همه تنظیمات در گزینه واحد `fip_settings`.
- پیاده‌سازی Settings API برای بخش‌های زیر:
  - تنظیمات صفحات پنل.
  - تنظیمات اطلاعیه داشبورد.
  - تنظیمات حوزه‌های فعالیت.
  - تنظیمات پایه درخواست‌ها.
  - تنظیمات پایه پیامک sms.ir.
- sanitize و validation اولیه برای همه تنظیمات.
- متدهای کمکی در `FIP_Settings`:
  - `get_settings()`
  - `get_option( $key, $default = null )`
  - `get_page_url( $key )`
  - `get_activity_fields()`
  - `sanitize_settings( $input )`
  - `render_settings_page()`
- ثبت شورت‌کدهای پنل و بارگذاری قالب متناظر از پوشه `templates/`.
- enqueue شدن assetهای فرانت‌اند فقط هنگام render شدن شورت‌کد.
- enqueue شدن assetهای ادمین فقط در صفحه تنظیمات افزونه.
- استایل ساده و RTL-friendly برای صفحه تنظیمات و placeholderهای فرانت‌اند.


### Phase 2

- فعال شدن شورت‌کد `[filter_portal_login]` برای ورود/ثبت‌نام با موبایل و OTP.
- پذیرش فرمت‌های `09123456789`، `9123456789`، `+989123456789` و `989123456789` و ذخیره داخلی شماره به شکل `09xxxxxxxxx`.
- تولید کد ۵ رقمی با اعتبار ۳ دقیقه، مصرف یک‌باره و ذخیره فقط به‌صورت hash در transientهای وردپرس.
- محدودیت امنیتی ارسال: حداقل فاصله ۶۰ ثانیه برای ارسال مجدد، حداکثر ۳ ارسال برای هر موبایل در ۱۰ دقیقه و حداکثر ۱۰ ارسال برای هر IP در ۱۰ دقیقه.
- محدودیت تلاش ناموفق: حداکثر ۵ تلاش اشتباه برای هر کد؛ پس از رسیدن به سقف، کد حذف می‌شود و کاربر باید کد جدید بگیرد.
- ارسال mock بدون هیچ API call واقعی: آخرین کد تست در transient با کلید `fip_last_mock_otp_{mobile_hash}` ذخیره می‌شود.
- نمایش کد تست فقط برای مدیران دارای `manage_options` یا زمانی که `WP_DEBUG` فعال است؛ کاربران عادی در محیط production کد mock را نمی‌بینند.
- جستجوی کاربر بعد از تأیید OTP در متاهای `fip_mobile`، `billing_phone`، `mobile` و `phone` بدون افشای وجود یا نبود حساب قبل از تأیید کد.
- جلوگیری امن از ورود در صورت وجود چند کاربر با یک شماره موبایل و نمایش پیام عمومی پشتیبانی.
- ساخت خودکار کاربر جدید بدون نیاز به ایمیل واقعی، با نام کاربری `fip_09xxxxxxxxx`، ایمیل محلی `09xxxxxxxxx@no-email.local`، رمز تصادفی قوی و نقش `customer` در صورت فعال بودن WooCommerce یا `subscriber` در غیر این صورت.
- ذخیره متاهای `fip_mobile`، `fip_mobile_verified` و `fip_profile_completed` برای کاربران ایجادشده یا واردشده.
- ریدایرکت پایه پس از ورود: اگر پروفایل ناقص باشد به صفحه تکمیل پروفایل و اگر کامل باشد به داشبورد یا `redirect_to` امن.
- افزودن helperهای احراز هویت برای URLهای ورود، داشبورد، تکمیل پروفایل، logout و پیام‌های نیاز به ورود/تکمیل پروفایل.


### Phase 4

- پیاده‌سازی ماژول `FIP_Profile` برای خواندن، اعتبارسنجی، تکمیل و ذخیره پروفایل کاربر.
- فعال شدن شورت‌کدهای `[filter_portal_complete_profile]` و `[filter_portal_edit_profile]` با فرم فارسی، راست‌به‌چپ و امن.
- ذخیره کلیدهای متای `fip_first_name`، `fip_last_name`، `fip_company`، `fip_activity_field`، `fip_position`، `fip_province`، `fip_city`، `fip_birth_date` و `fip_profile_completed`.
- همگام‌سازی `first_name` و `last_name` وردپرس پس از ذخیره موفق پروفایل.
- اعتبارسنجی فیلدهای الزامی، حوزه فعالیت، استان و رابطه شهر با استان.
- نمایش شماره موبایل از متای `fip_mobile` به‌صورت read-only و بدون امکان ویرایش در فرم پروفایل.
- بارگذاری داده شهرها در JavaScript فقط روی صفحات دارای فرم پروفایل، بدون AJAX.

### Phase 6

- ثبت Custom Post Type خصوصی `fip_request` برای درخواست‌های استعلام مشتریان با UI پایه در پیشخوان وردپرس.
- استفاده از post meta به‌جای post status سفارشی برای وضعیت درخواست‌ها.
- تعریف وضعیت‌های مجاز درخواست:
  - `pending`: در انتظار بررسی
  - `reviewing`: در حال بررسی
  - `need_info`: نیاز به اطلاعات بیشتر
  - `answered`: پاسخ داده شد
  - `rejected`: رد شد
  - `closed`: بسته شد
- آماده‌سازی کلیدهای متای مرکزی برای شماره درخواست، کاربر، موبایل، وضعیت، آیتم‌ها، یادداشت عمومی، پاسخ مدیر، یادداشت داخلی، به‌روزرسانی‌های مشتری و وضعیت آخرین SMS.
- اضافه شدن متدهای کمکی در `FIP_Requests` برای تولید شماره درخواست، ایجاد درخواست داخلی، خواندن درخواست، دریافت درخواست‌های یک کاربر، بررسی مالکیت و ثبت به‌روزرسانی مشتری.
- اضافه شدن ستون‌های ساده پیشخوان برای شماره درخواست، کاربر، موبایل، وضعیت، تعداد آیتم‌ها و تاریخ.
- آماده‌سازی ساختار داده بدون فرم فرانت‌اند، بدون جستجوی محصول، بدون UI مدیریت پیشرفته و بدون ارسال پیامک.

### Phase 1.5

- جایگزینی تنظیمات پیامک قبلی با تنظیمات اختصاصی sms.ir.
- آماده‌سازی فیلدهای API Key، شماره خط پیش‌فرض و Template IDهای sms.ir.
- آماده‌سازی کلاس provider با نام `FIP_SMSIR_Provider` بدون ارسال واقعی پیامک.
- حفظ option اصلی `fip_settings` و نگهداری سازگاری با تنظیمات ذخیره‌شده قبلی بدون ایجاد notice.

موارد زیر عمداً هنوز پیاده‌سازی نشده‌اند و باید در فازهای بعدی اضافه شوند:

- تولید، ارسال و اعتبارسنجی OTP.
- اتصال واقعی به API سامانه sms.ir.
- ارسال پیامک تست.
- جستجوی AJAX محصولات WooCommerce.
- ثبت فرم درخواست.
- redirectهای احراز هویت.
- پنل مدیریت درخواست‌ها.
- جدول یا گزارش لاگ پیامک‌ها.

## راهنمای نصب افزونه روی وردپرس

1. پوشه افزونه را با نام `filter-inquiry-portal` داخل مسیر زیر قرار دهید:

   ```text
   wp-content/plugins/filter-inquiry-portal/
   ```

2. وارد پیشخوان وردپرس شوید.
3. از مسیر **افزونه‌ها**، افزونه **Filter Inquiry Portal** را فعال کنید.
4. پس از فعال‌سازی، در منوی مدیریت گزینه **استعلام فیلتر** را مشاهده کنید.
5. وارد صفحه تنظیمات شوید و صفحات پنل و تنظیمات پایه را انتخاب یا ذخیره کنید.

## راهنمای ساخت صفحات و قراردادن شورت‌کدها

برای هر بخش پنل یک برگه وردپرس بسازید و شورت‌کد مربوط به همان بخش را داخل محتوای برگه قرار دهید. سپس از مسیر **استعلام فیلتر** در پیشخوان، هر برگه را در فیلد متناظر انتخاب کنید.

| داشبورد | `[filter_portal_dashboard]` | صفحه داشبورد |

| ویرایش پروفایل | `[filter_portal_edit_profile]` | صفحه ویرایش پروفایل |

| درخواست‌های من | `[filter_portal_my_requests]` | صفحه درخواست‌های من |
| جزئیات درخواست | `[filter_portal_request_detail]` | صفحه جزئیات درخواست |

## راهنمای تنظیمات افزونه

صفحه تنظیمات از منوی **استعلام فیلتر** در پیشخوان وردپرس در دسترس است و فقط کاربران دارای capability `manage_options` می‌توانند آن را مدیریت کنند.

### تنظیمات صفحات پنل

در این بخش باید برگه‌های ساخته‌شده را به بخش‌های افزونه متصل کنید. هر dropdown لیست برگه‌های وردپرس را نمایش می‌دهد و مقدار ذخیره‌شده، ID همان برگه است.

### تنظیمات اطلاعیه داشبورد

- **فعال بودن اطلاعیه داشبورد:** ذخیره می‌کند که اطلاعیه در فازهای بعدی نمایش داده شود یا خیر.
- **عنوان اطلاعیه:** عنوان کوتاه اطلاعیه.
- **متن اطلاعیه:** متن کامل اطلاعیه برای استفاده در داشبورد آینده.

### راهنمای تنظیم حوزه‌های فعالیت

در فیلد **حوزه‌های فعالیت** هر گزینه را در یک خط جداگانه وارد کنید. مقدار پیش‌فرض:

```text
تولیدکننده ماشین‌آلات سنگین
راه‌آهن و ریلی
نفت و پتروشیمی
کمپرسور و ژنراتور
معادن
```

این لیست در فرم‌های تکمیل و ویرایش پروفایل مشتری استفاده می‌شود. اگر تنظیمات خالی باشد، فرم پروفایل از همین مقادیر پیش‌فرض استفاده می‌کند.

### تنظیمات درخواست‌ها

- حداکثر تعداد آیتم در هر درخواست: عدد ۱ تا ۱۰.
- حداکثر تعداد اطلاعات تکمیلی برای هر درخواست: عدد ۰ تا ۱۰.
- حداکثر حجم تصویر، مگابایت: عدد ۱ تا ۱۰.
- ایمیل مدیر برای اطلاع‌رسانی درخواست‌ها: باید ایمیل معتبر باشد؛ در غیر این صورت ایمیل مدیر سایت ذخیره می‌شود.
- ارسال پیامک به مدیر هنگام ثبت درخواست جدید: فقط ذخیره می‌شود و هنوز ارسال واقعی انجام نمی‌شود.
- شماره موبایل مدیر: sanitize می‌شود ولی در این فاز اعتبارسنجی عمیق ندارد.

### تنظیمات sms.ir

در این بخش تنظیمات موردنیاز برای اتصال آینده به سامانه sms.ir ذخیره می‌شود:

- فعال‌سازی پیامک.
- API Key سامانه sms.ir.
- شماره خط پیش‌فرض، فقط برای ارسال‌های غیر Verify.
- استفاده از ارسال قالبی/Verify برای پیامک‌های اطلاع‌رسانی.
- Template ID کد ورود.
- Template ID ثبت درخواست.
- Template ID وضعیت در حال بررسی.
- Template ID وضعیت نیاز به اطلاعات بیشتر.
- Template ID وضعیت پاسخ داده شد.
- Template ID وضعیت رد شد.
- Template ID وضعیت بسته شد.
- Template ID پیامک مدیر برای درخواست جدید.
- شماره تست پیامک.

افزونه برای پیامک‌های ورود و اطلاع‌رسانی‌های خدماتی از معماری Verify/template سامانه sms.ir استفاده خواهد کرد. API Key و Template IDها در `fip_settings` ذخیره می‌شوند، اما ارسال واقعی پیامک تا فاز اتصال SMS پیاده‌سازی نشده است و در Phase 1.5 هیچ API call واقعی انجام نمی‌شود.

## راهنمای ورود موبایلی و OTP mock

فرم `[filter_portal_login]` در دو مرحله کار می‌کند:

1. کاربر شماره موبایل ایرانی خود را وارد می‌کند و روی **دریافت کد ورود** می‌زند.
2. افزونه شماره را normalize و validate می‌کند، محدودیت‌های ارسال را بررسی می‌کند، یک OTP پنج‌رقمی می‌سازد و آن را به‌صورت mock ذخیره می‌کند.
3. کاربر کد را در مرحله دوم وارد می‌کند و روی **ورود / ثبت‌نام** می‌زند.
4. پس از تأیید کد، افزونه کاربر موجود را با شماره موبایل پیدا می‌کند یا کاربر جدید می‌سازد، کاربر را با cookie وردپرس وارد می‌کند و بر اساس وضعیت پروفایل ریدایرکت انجام می‌دهد.

### رفتار mock OTP

در این فاز هیچ پیامک واقعی ارسال نمی‌شود؛ فعال یا غیرفعال بودن تنظیمات SMS و ناقص بودن API Key یا Template IDهای sms.ir مانع کارکرد ورود OTP نیست. آخرین کد تولیدشده برای هر موبایل فقط برای توسعه در transient ذخیره می‌شود و در فرم ورود صرفاً برای مدیران سایت یا در حالت `WP_DEBUG` نمایش داده می‌شود. کاربران عادی در محیط production نباید کد تست را ببینند. اتصال واقعی به sms.ir در **Phase 3** انجام خواهد شد.

### محدودیت‌های امنیتی OTP

- کد فقط ۵ رقم عددی است.
- اعتبار هر کد ۳ دقیقه است.
- کد پس از استفاده موفق حذف می‌شود و یک‌بارمصرف است.
- برای هر کد حداکثر ۵ تلاش اشتباه مجاز است.
- ارسال مجدد برای هر موبایل پس از ۶۰ ثانیه مجاز می‌شود.
- برای هر موبایل حداکثر ۳ ارسال در ۱۰ دقیقه مجاز است.
- برای هر IP حداکثر ۱۰ ارسال در ۱۰ دقیقه مجاز است.
- کد خام در transient اصلی OTP ذخیره نمی‌شود و فقط hash آن نگهداری می‌شود.

### چک‌لیست تست Phase 2

- شماره موبایل نامعتبر رد می‌شود.
- شماره موبایل معتبر کد mock دریافت می‌کند.
- محدودیت ارسال مجدد ۶۰ ثانیه‌ای عمل می‌کند.
- کد اشتباه تعداد تلاش‌ها را افزایش می‌دهد.
- بیش از ۵ تلاش اشتباه کد را نامعتبر می‌کند.
- کد صحیح کاربر را وارد سایت می‌کند.
- برای شماره جدید کاربر جدید ساخته می‌شود.
- برای شماره موجود همان کاربر دوباره استفاده می‌شود.
- وجود شماره تکراری بین چند کاربر ورود را به‌صورت امن مسدود می‌کند.
- پروفایل ناقص به صفحه تکمیل پروفایل ریدایرکت یا لینک می‌شود.
- پروفایل کامل به داشبورد ریدایرکت یا لینک می‌شود.
- کاربران عادی کد mock را نمی‌بینند مگر شرط مجاز `WP_DEBUG`/مدیر برقرار باشد.

## راهنمای تنظیم صفحات پنل

1. از مسیر **برگه‌ها > افزودن برگه**، برگه‌های لازم را بسازید.
2. داخل هر برگه فقط شورت‌کد متناظر را قرار دهید.
3. برگه را منتشر کنید.
4. به صفحه **استعلام فیلتر** بروید.
5. در بخش **تنظیمات صفحات پنل**، برای هر بخش برگه مرتبط را انتخاب کنید.
6. تنظیمات را ذخیره کنید.

## راهنمای بهره‌برداری روی سایت

### چگونه صفحات پنل را بسازند

اپراتور سایت باید برای هر بخش پنل یک برگه جداگانه ایجاد کند؛ برای نمونه «ورود پنل استعلام»، «داشبورد پنل استعلام»، «ثبت درخواست فیلتر» و «درخواست‌های من».

### هر شورت‌کد در چه صفحه‌ای قرار بگیرد

- صفحه ورود: `[filter_portal_login]`
- صفحه داشبورد: `[filter_portal_dashboard]`
- صفحه تکمیل پروفایل: `[filter_portal_complete_profile]`
- صفحه ویرایش پروفایل: `[filter_portal_edit_profile]`
- صفحه ثبت درخواست: `[filter_portal_new_request]`
- صفحه درخواست‌های من: `[filter_portal_my_requests]`
- صفحه جزئیات درخواست: `[filter_portal_request_detail]`

### چگونه صفحات را از تنظیمات افزونه انتخاب کنند

پس از ساخت و انتشار برگه‌ها، از منوی **استعلام فیلتر** وارد صفحه تنظیمات شوید و در بخش **تنظیمات صفحات پنل**، هر برگه را از dropdown مربوط انتخاب کنید. سپس روی **ذخیره تنظیمات** کلیک کنید.

### چگونه اطلاعیه داشبورد را فعال کنند

در بخش **تنظیمات اطلاعیه داشبورد**، گزینه **فعال بودن اطلاعیه داشبورد** را فعال کنید، عنوان و متن اطلاعیه را وارد کنید و تنظیمات را ذخیره کنید. نمایش عملی این اطلاعیه در داشبورد واقعی در فازهای بعدی کامل می‌شود.

### چگونه حوزه‌های فعالیت را ویرایش کنند

در بخش **تنظیمات پروفایل**، فیلد **حوزه‌های فعالیت** را ویرایش کنید. هر گزینه باید در یک خط جداگانه باشد. این گزینه‌ها در فرم‌های تکمیل و ویرایش پروفایل نمایش داده می‌شوند. اگر تنظیمات خالی باشد، افزونه از گزینه‌های پیش‌فرض «تولیدکننده ماشین‌آلات سنگین»، «راه‌آهن و ریلی»، «نفت و پتروشیمی»، «کمپرسور و ژنراتور» و «معادن» استفاده می‌کند.

### راهنمای فرم‌های پروفایل

برای صفحه تکمیل پروفایل، شورت‌کد `[filter_portal_complete_profile]` و برای صفحه ویرایش پروفایل، شورت‌کد `[filter_portal_edit_profile]` را داخل برگه‌های مربوط قرار دهید. کاربر خارج‌شده فقط پیام «برای تکمیل حساب کاربری ابتدا وارد شوید.» را می‌بیند و در این فاز redirect سراسری انجام نمی‌شود. کاربر واردشده فرم را با فیلدهای زیر مشاهده می‌کند:

- `نام`، `نام خانوادگی`، `حوزه فعالیت`، `استان` و `شهر` الزامی هستند.
- `نام شرکت / فروشگاه`، `سمت شما` و `تاریخ تولد` اختیاری هستند.
- `شماره موبایل` از user meta با کلید `fip_mobile` خوانده می‌شود و فقط برای نمایش است؛ این فرم موبایل را ذخیره یا ویرایش نمی‌کند.
- پس از ذخیره موفق، `fip_profile_completed` برابر `1` می‌شود و نام/نام خانوادگی وردپرس نیز به‌روزرسانی می‌شود.

### منبع استان‌ها و شهرها

فهرست استان‌ها و شهرها در فایل `includes/data/iran-cities.php` به‌صورت آرایه PHP نگهداری می‌شود، نه JSON. فرم پروفایل همین داده را برای select استان و شهر استفاده می‌کند و برای به‌روزرسانی city select در فرانت‌اند، داده فقط روی صفحات دارای فرم پروفایل به JavaScript تزریق می‌شود. اعتبارسنجی سمت سرور نیز بررسی می‌کند که استان وجود داشته باشد و شهر انتخابی متعلق به همان استان باشد.

### فعلاً چه چیزهایی هنوز کار نمی‌کند

ساختار ذخیره‌سازی درخواست‌ها آماده است و فرم پروفایل کار می‌کند، اما همچنان ورود/OTP، فرم ثبت درخواست فرانت‌اند، جستجوی محصول WooCommerce، مدیریت پیشرفته درخواست‌ها و ارسال واقعی پیامک وجود ندارد. ورود عملیاتی، پیامک عملیاتی و جریان کامل مشتری در فازهای بعدی تکمیل می‌شوند.

### هشدار بهره‌برداری

این نسخه برای آماده‌سازی ساختار، تنظیمات، فرم پروفایل، معماری پیامک sms.ir و مدل داده درخواست‌ها است. **ورود OTP، فرم ثبت درخواست، جستجوی محصول، ارسال پیامک و گردش کامل پاسخ‌دهی در فازهای بعدی فعال می‌شوند**؛ بنابراین نباید انتظار عملکرد کامل پنل مشتری در سایت نهایی را از Phase 4 داشت.

### مدل داده درخواست‌ها در Phase 6

درخواست‌ها با post type خصوصی `fip_request` ذخیره می‌شوند. این post type عمومی نیست، archive و rewrite ندارد و در REST API نیز expose نشده است. UI آن فقط برای مشاهده پایه در پیشخوان فعال است.

کلیدهای متای آماده‌شده برای فازهای بعدی:

- `fip_request_number`
- `fip_user_id`
- `fip_user_mobile`
- `fip_status`
- `fip_items`
- `fip_items_count`
- `fip_general_note`
- `fip_admin_response`
- `fip_internal_note`
- `fip_customer_updates`
- `fip_last_sms_status`
- `fip_last_sms_sent_at`

متدهای اصلی `FIP_Requests`:

- `get_statuses()` برای دریافت label وضعیت‌ها.
- `get_meta_keys()` برای دریافت کلیدهای متای مرکزی.
- `generate_request_number( $post_id )` برای ساخت شماره‌ای مانند `FQ-000123`.
- `create_request( $user_id, array $items = array(), $general_note = '' )` برای ایجاد داخلی درخواست.
- `get_user_requests( $user_id, $paged = 1, $per_page = 10 )` برای دریافت query درخواست‌های کاربر.
- `get_request( $request_id )` برای دریافت آرایه ساختاریافته درخواست.
- `user_can_view_request( $user_id, $request_id )` برای بررسی مالکیت یا دسترسی مدیر.
- `get_customer_updates( $request_id )` و `add_customer_update( $request_id, $user_id, $note = '', $image_id = 0 )` برای آماده‌سازی به‌روزرسانی‌های آینده مشتری.

نکته امنیتی: متد `get_request()` داده کامل داخلی از جمله `internal_note` را برمی‌گرداند و برای خروجی مستقیم فرانت‌اند طراحی نشده است. هنگام ساخت UI مشتری در فازهای بعدی نباید یادداشت داخلی مدیر به کاربر نمایش داده شود.

### استانداردهای نام‌گذاری

- گزینه‌ها، توابع، هوک‌ها، nonceها، actionها و meta keyها باید با پیشوند `fip_` باشند.
- کلاس‌ها باید با پیشوند `FIP_` باشند.
- option اصلی تنظیمات `fip_settings` است.
- handle assetهای افزونه `fip_frontend` و `fip_admin` هستند.
- فایل‌های قالب در پوشه `templates/` نگهداری می‌شوند.

### نکات امنیتی

- دسترسی صفحه تنظیمات با `manage_options` کنترل می‌شود.
- تنظیمات با Settings API ذخیره می‌شوند.
- همه مقادیر ورودی sanitize می‌شوند.
- همه خروجی‌های قالب و ادمین escape می‌شوند.
- فیلدهای حساس پیامک با input نوع `password` نمایش داده می‌شوند.
- فایل‌های PHP با محافظ `ABSPATH` اجرا می‌شوند.
- فرم‌های پروفایل از nonce `fip_profile_nonce` استفاده می‌کنند و همه ورودی‌ها sanitize و همه خروجی‌ها escape می‌شوند.
- در این نسخه هیچ پردازش عملی فرم ثبت درخواست، جستجوی محصول، ارسال پیامک یا UI مدیریت پیشرفته درخواست انجام نمی‌شود.
- خروجی ستون‌های ادمین برای درخواست‌ها escape می‌شود و مقادیر قبل از ذخیره sanitize می‌شوند.

### نکات پرفورمنس

- assetهای فرانت‌اند به‌صورت سراسری enqueue نمی‌شوند و فقط هنگام render شدن شورت‌کدها لود می‌شوند؛ داده استان/شهر فقط در شورت‌کدهای پروفایل به JavaScript داده می‌شود.
- assetهای ادمین فقط در صفحه تنظیمات افزونه لود می‌شوند.
- تنظیمات در یک option آرایه‌ای ذخیره می‌شوند تا بار اضافی روی دیتابیس کم باشد.
- افزونه در این فاز جدول سفارشی، AJAX عملیاتی یا query فرانت‌اند سنگین ندارد؛ درخواست‌ها با CPT خصوصی و post meta ذخیره می‌شوند.
